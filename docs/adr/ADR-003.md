# ADR-003 — Stack Técnica do Backend

## Status

Aceito

---

# Contexto

O Supervisor AI está iniciando o desenvolvimento do MVP e será mantido inicialmente por uma única pessoa.

A solução precisa ser simples de executar e manter, sem comprometer a separação entre conectores, Motor de Importação, Motor de Processamento, Motor de Regras e API.

O contrato técnico e as credenciais da API do MK Solutions ainda não estão disponíveis. Por isso, a primeira fonte de dados utilizada para validar o fluxo de importação poderá ser composta por arquivos CSV e JSON.

---

# Problema

É necessário definir uma stack mínima que:

- preserve a arquitetura baseada em motores;
- permita adicionar integrações futuras;
- ofereça migrações e testes desde o início;
- seja simples para desenvolvimento e operação do MVP;
- não introduza a complexidade de microserviços ou processamento distribuído antes de existir necessidade comprovada;
- permita validar o Motor de Importação sem depender de acesso real ao MK Solutions.

---

# Decisão

## Arquitetura da aplicação

O backend será desenvolvido como um monólito modular.

Os componentes serão executados e implantados como uma única aplicação, mantendo separação interna por responsabilidade técnica.

A organização deverá preservar, no mínimo, as seguintes responsabilidades:

- conectores;
- Motor de Importação;
- Motor de Processamento;
- Motor de Regras;
- API;
- serviços;
- banco de dados.

Não serão adotados microserviços no MVP.

## Stack do backend

A stack adotada será:

- Python 3.12;
- FastAPI;
- SQLAlchemy 2.x;
- psycopg 3;
- PostgreSQL 16;
- Alembic para migrações;
- pytest para testes;
- Ruff para lint e formatação;
- uv para gerenciamento de dependências e ambiente Python.

As operações de banco de dados serão síncronas inicialmente.

O PostgreSQL 16 será a versão principal suportada inicialmente. Atualizações
para uma nova versão principal deverão ser precedidas por validação de
compatibilidade da aplicação, das migrações e dos dados existentes.

O uso de operações assíncronas somente deverá ser considerado quando houver uma necessidade concreta, sustentada por medições ou pelas características confirmadas de uma integração.

## Ambiente local

O Docker Compose será utilizado somente para executar o PostgreSQL no ambiente local.

As credenciais publicadas no `.env.example` destinam-se exclusivamente ao
desenvolvimento local. Elas não deverão ser reutilizadas em ambientes de
homologação ou produção.

A aplicação Python será executada diretamente no ambiente gerenciado pelo uv.

Outros componentes de infraestrutura, como filas, workers distribuídos, Redis ou orquestradores, não farão parte da fundação inicial.

## Primeira integração simulada

O primeiro fluxo de importação será validado com arquivos CSV e JSON.

Essa implementação será tratada como um conector de arquivos ou fonte simulada genérica. Ela deverá utilizar o mesmo contrato esperado pelo Motor de Importação para conectores futuros.

Nenhuma integração, classe, pacote ou conector será denominado MK enquanto não existirem contrato técnico, credenciais e amostras reais que confirmem o formato da integração.

As referências ao MK Solutions na documentação de negócio continuam representando o sistema oficial e uma integração futura, não um contrato técnico já definido.

---

# Consequências

## Benefícios

- Baixa complexidade operacional para o MVP.
- Um único processo de aplicação para desenvolver, testar e implantar.
- Separação interna compatível com a arquitetura lógica existente.
- Stack adequada para importação, validação e processamento de dados.
- Evolução controlada do banco por migrações versionadas.
- Possibilidade de testar o fluxo de importação antes da disponibilidade de APIs externas.
- Substituição futura da fonte simulada por conectores reais sem alterar as responsabilidades do Motor de Importação.
- Menor risco de consolidar prematuramente suposições incorretas sobre o MK Solutions.

## Trade-offs

- O PostgreSQL continuará sendo um serviço externo necessário no desenvolvimento local.
- O monólito exigirá disciplina para preservar as fronteiras entre os componentes internos.
- Operações síncronas podem precisar ser revistas se integrações futuras apresentarem grande volume de chamadas concorrentes.
- A fonte simulada valida o fluxo técnico, mas não valida autenticação, limites, disponibilidade ou formato da futura API do MK Solutions.

---

# Critérios para revisão

Esta decisão poderá ser revista quando:

- medições demonstrarem que operações síncronas são insuficientes;
- uma integração exigir processamento assíncrono ou filas duráveis;
- o volume ou a operação da plataforma justificarem separação de processos;
- limitações técnicas confirmadas de uma fonte externa exigirem adaptação da stack.

Qualquer revisão deverá ser registrada em uma nova ADR.
